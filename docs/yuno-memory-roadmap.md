# ゆの 記憶・会話仕様ロードマップ

このメモは、Discord bot「唯乃 / ゆの」の今後の記憶・会話設計の方針をまとめるものです。  
直近の実装指示ではなく、今後の変更で判断がぶれないようにするための設計メモです。

## 基本方針

ゆのは、安全性を最大化する管理botではなく、自然に会話のそばにいる存在として設計する。

そのため、記憶や自動返信の仕様も、過度に制限するより、次のことを重視する。

- 何が起きたか分かる
- 後から戻せる
- 会話の自然さを壊さない
- 実装の都合が前面に出ない

記憶変更についても、完全に禁止するより、明示・履歴・undoによって扱える状態にする。

## 現在の記憶仕様

現在の長期記憶は records v4 を標準とする。

現在の主な分類は以下。

- `memory`
  - 本人について覚えておく内容
  - 安定した事実、関心、継続中の活動、継続的な好み
- `interaction_preference`
  - ゆのの話し方・扱い方への希望
  - 呼び方、返答態度、避けたい言い方など

この分類は現状では有効だが、今後の本命の分類軸ではない可能性がある。

## 今後重要になる分類軸

本当に必要な分類は、内容の種類よりも、その記憶がどの範囲に効くかである。

今後は以下の scope を意識する。

- `user`
  - 特定ユーザー個人に属する記憶
  - 例: 関心、制作活動、継続的な好み、呼び方
- `guild`
  - サーバー全体に属する記憶
  - 例: このサーバーでのゆのの距離感、ノリ、振る舞い
- `channel`
  - チャンネル単位に属する記憶・振る舞い
  - 例: このチャンネルでは自動返信を控える、作業ログには反応しすぎない
- `context`
  - その場限りの会話文脈
  - 長期記憶には入れない
- `inner`
  - ゆの側の余韻・受け取り方
  - 記憶ではないが、会話の声に影響する

今の `memory / interaction_preference` は、将来的には scope の下にある副分類として扱う方が自然かもしれない。

例:

```json
{
  "scope": "user",
  "record_type": "memory",
  "item": "フラクタルに関心がある"
}
```

```json
{
  "scope": "guild",
  "record_type": "behavior",
  "item": "このサーバーでは非メンション反応を控えめにする"
}
```

```json
{
  "scope": "channel",
  "record_type": "behavior",
  "item": "このチャンネルでは作業ログへの反応を控える"
}
```

## 保存先の考え方

今後は、すべてを個人の longterm_memory に入れるのではなく、保存範囲ごとに分けることを検討する。

現状:

- `longterm_memory.json`
  - 個人記憶
- `guild_notes.json`
  - サーバーメモ
- `auto_reply_settings.json`
  - 自動返信設定
- `chat_history`
  - 会話履歴
- `inner_log`
  - ゆの側の余韻

今後の方向:

- 個人に関することは user memory
- サーバーの空気や振る舞いは guild memory / guild behavior
- チャンネル固有の振る舞いは channel behavior
- 自動返信ON/OFFのような機械的設定は auto_reply_settings
- 一時的な流れは長期記憶に入れない

## 衝突時の考え方

記憶や設定が衝突した場合、基本的には狭い範囲の情報を優先する。

おおまかな優先順位:

```text
channel > guild > user > global persona
```

ただし、呼び方や本人の好みなど、明らかに個人に属するものは user を優先してよい。

重要なのは、単純な分類名ではなく「その情報がどの場に属するか」を見ること。

## 自動記憶の考え方

自動記憶では、`add_item` / `delete_item` / `rewrite_item` / `set_slot` / `delete_slot` を扱ってよい。

理由:

- 記憶変更はリアクションで明示される
- `/memory recent` で確認できる
- `/memory undo` で戻せる
- 自然な会話では、明確な訂正や撤回に反応して記憶を書き換えられる方がよい

ただし、以下は守る。

- 曖昧な削除や広範囲整理はしない
- 対象が特定できない場合は返答で短く確認する
- 記憶変更に失敗したのに「覚えた」ように言わない
- 記憶変更の明示とundo導線を維持する

## 会話の自然さについて

ゆのは、実装仕様を説明するbotではなく、会話のそばにいる存在として振る舞う。

そのため、promptや返答では以下を避ける。

- 実装用語が前面に出る
- 毎回「記憶しました」と言いすぎる
- 反応絵文字を機械的に大量につける
- 記憶管理が会話の主役になる
- 安全確認が多すぎて会話が止まる

通常reactionは最大5個まで使えるままにする。  
ただし普段は控えめにし、ユーザーが明示した場合や複数絵文字で表現するのが自然な場合だけ多めに使う。

## inner の扱い

`inner` は単なる方針メモではなく、ゆのらしさに影響している可能性がある。

そのため、安易に削ったり、機械的な方針メモだけに寄せたりしない。

ただし、inner が自己演出として強くなりすぎると、会話内容よりゆの自身の雰囲気が前に出る可能性がある。

今後触る場合は、次の方針にする。

- inner は残してよい
- 詩的さや余韻を完全には消さない
- ただし会話内容より自分の雰囲気を優先しない
- inner_log の保存処理は慎重に扱う

## 複数返答・継続発話について

将来的に、ゆのの返答を短くし、複数回に分けて返す仕組みを検討する。

ただし、単に `replies` 配列で最初から複数文を生成するだけでは不十分。

本当にやりたいのは、次のような継続発話ループである。

1. まず短く返す
2. 自分が送った返答を読む
3. まだ言うべきことがあるか判断する
4. 必要なら次の短い返答を送る
5. ユーザーが途中で返したら止まる

これは自然さに効くが、API呼び出し・遅延・連投感の問題があるため、今回はまだ実装しない。

## 近い将来の整理候補

優先度が高いもの:

1. memory_operations の説明を短くし、実装っぽさを減らす
2. 非メンションpolicyを日本語で短くする
3. `record_flag` を `history_flag` などに改名する
4. 到達しないDMコマンド分岐を削る
5. 記憶変更に失敗したとき、返答内容と実際の保存状態がズレないようにする
6. 非メンション文脈反応の直前ゆの判定を、表示名ではなくbot自身のIDベースにする
7. user / guild / channel の scope 設計を検討する

今回はやらないもの:

- scope付き記憶の本実装
- 複数返答 / 継続発話ループ
- quiet mode
- auto_reply のmode化
- memory_model の大規模再整理
- `/memory` UIの大改造

## 今後の判断基準

変更に迷ったら、次を見る。

- それは会話を自然にするか
- ゆのの存在感が強くなりすぎないか
- 記憶管理が前面に出すぎないか
- 後から確認・undoできるか
- 個人記憶とサーバー/チャンネル記憶を混ぜていないか
- 実装の都合をゆのの人格に押し出していないか

ゆのは、記憶管理botではなく、会話のそばに残る存在として設計する。
